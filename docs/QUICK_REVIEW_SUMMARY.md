## 🎯 역할별 10초 요약

### PO / 기획자 (PRD.md)

**왜 중요한가?**: 모든 기능 개발의 기준이 되는 문서

**핵심 검토 항목**:
1. ✅ **비전**: "독서 능력 진단 및 개선 플랫폼" 맞는가?
2. ✅ **사용자**: 학생/학부모/교사/연구원/관리자 6명이 맞는가?
3. ✅ **기능**: 8개 기능 영역이 모두 필요한가?
4. ✅ **우선순위**: P0/P1/P2 분류가 현실적인가?
5. ✅ **성공 지표**: KPI 5개가 측정 가능한가?

**검토 시간**: 1-2시간
**체크리스트**: REVIEW_GUIDE.md의 "PO / 기획자" 섹션 참조

---

### 기술리더 / 아키텍트 (TRD.md)

**왜 중요한가?**: 모든 기술적 결정의 근거가 되는 문서

**핵심 검토 항목**:
1. ✅ **아키텍처**: 4층 구조 (Presentation-Application-Domain-Infrastructure) 적절한가?
2. ✅ **데이터베이스**: 31개 테이블 설계가 정규화되고 관계 정의가 완전한가?
3. ✅ **API 설계**: RESTful /api/v1/ 구조와 50+ 엔드포인트가 현실적인가?
4. ✅ **서비스 레이어**: 비즈니스 로직 분리 패턴이 명확한가?
5. ✅ **마이그레이션**: 9개 누락 모델 복원 + 관계 수정 계획이 구체적인가?

**검토 시간**: 2-3시간
**체크리스트**: REVIEW_GUIDE.md의 "기술리더 / 아키텍트" 섹션 참조

---

### 백엔드 개발자 (API_SPECIFICATION.md + DATABASE_SCHEMA.md)

**왜 중요한가?**: 실제 코드 구현의 기준이 되는 문서

**API 검토** (API_SPECIFICATION.md):
1. ✅ **엔드포인트**: 모든 CRUD 작업 (Create/Read/Update/Delete) 정의되어 있는가?
2. ✅ **요청/응답**: JSON 스키마가 명확하고 예제가 충분한가?
3. ✅ **에러 처리**: HTTP 상태 코드와 에러 메시지가 정의되었는가?
4. ✅ **특수 기능**: AI 피드백, 리포트 생성 엔드포인트가 명확한가?

**Database 검토** (DATABASE_SCHEMA.md):
1. ✅ **테이블**: 31개 테이블 정의가 완전하고 필드가 적절한가?
2. ✅ **관계**: 외래키와 조인 테이블 정의가 정확한가?
3. ✅ **인덱스**: 성능 최적화를 위한 인덱스가 충분한가?
4. ✅ **제약 조건**: NOT NULL, UNIQUE, CHECK 제약이 적절하게 정의되었는가?

**검토 시간**: 3-5시간
**체크리스트**: REVIEW_GUIDE.md의 "백엔드 개발자" 섹션 참조

---

### 전체 개발자 (DEVELOPER_GUIDE.md)

**왜 중요한가?**: 개발 환경 설정과 코드 표준의 기준

**핵심 검토 항목**:
1. ✅ **환경 설정**: `bundle install` → `bin/rails db:prepare` 절차가 명확한가?
2. ✅ **코드 표준**: RuboCop 규칙이 명확하고 실현 가능한가?
3. ✅ **Git 워크플로우**: Feature branch → PR → Main 프로세스가 명확한가?
4. ✅ **테스트 전략**: 단위/통합/시스템 테스트 작성 방법이 명확한가?

**검토 시간**: 1-2시간
**체크리스트**: REVIEW_GUIDE.md의 "개발자" 섹션 참조

---

### DevOps / SRE (DEPLOYMENT_GUIDE.md)

**왜 중요한가?**: 프로덕션 배포와 운영의 기준

**핵심 검토 항목**:
1. ✅ **배포 절차**: Railway 배포 자동화 및 manual step이 안전한가?
2. ✅ **모니터링**: 헬스 체크, 로깅, 에러 추적이 충분한가?
3. ✅ **백업/복구**: 데이터 손실 방지 및 복구 절차가 테스트되었는가?
4. ✅ **보안**: 환경 변수, HTTPS, 접근 제어가 완전한가?

**검토 시간**: 1-2시간
**체크리스트**: REVIEW_GUIDE.md의 "DevOps / SRE" 섹션 참조

---

### 구현팀 (MIGRATION_RUNBOOK.md)

**왜 중요한가?**: 기존 시스템 → 새 시스템 마이그레이션의 실행 계획

**핵심 검토 항목**:
1. ✅ **Phase 계획**: 7개 Phase (1주-2주)가 순서대로 정의되어 있는가?
2. ✅ **Step-by-Step**: 각 단계별 명확한 실행 절차가 있는가?
3. ✅ **검증**: 각 단계 후 검증 방법이 정의되었는가?
4. ✅ **롤백**: 문제 발생 시 롤백 계획이 명확한가?
5. ✅ **리소스**: 필요한 인원, 시간, 도구가 명시되었는가?

**검토 시간**: 1.5-2시간
**체크리스트**: REVIEW_GUIDE.md의 "구현팀" 섹션 참조

## 🔍 공통 검토 기준 (모든 역할)

모든 문서 검토 시 다음 기준을 적용하세요:

### 1단계: 이해 가능성 (5분)
- [ ] 문서를 읽고 전체 내용을 이해할 수 있는가?
- [ ] 기술 용어가 명확하게 정의되었는가?
- [ ] 예제나 다이어그램이 도움이 되는가?

### 2단계: 완전성 (10분)
- [ ] 필요한 모든 항목이 포함되었는가?
- [ ] 빠진 내용이 있는가?
- [ ] 중복되는 내용이 있는가?

### 3단계: 실현 가능성 (10분)
- [ ] 제시된 계획이 현실적인가?
- [ ] 필요한 리소스가 할당 가능한가?
- [ ] 예상 시간이 타당한가?

### 4단계: 일관성 (5분)
- [ ] 다른 문서와 일관성이 있는가?
- [ ] 용어, 정의, 네이밍이 일관적인가?
- [ ] 상충하는 내용이 있는가?

## ❓ 자주 묻는 질문 (FAQ)

**Q: 언제까지 검토해야 하나요?**
A: 2026-02-10까지 피드백을 FEEDBACK_TRACKER.md에 제출하세요.

**Q: 모든 내용을 다 읽어야 하나요?**
A: 역할에 따라 핵심 문서만 집중하세요. REVIEW_GUIDE.md의 역할별 체크리스트 참조.

**Q: 피드백은 어떻게 제출하나요?**
A: FEEDBACK_TRACKER.md의 해당 섹션에 작성 후 GitHub Issue로 통보하세요.

**Q: 다른 사람의 피드백과 충돌하면?**
A: 2026-02-11 피드백 통합 미팅에서 함께 논의합니다.

**Q: 검토 중 질문이 생기면?**
A: 아키텍트/리더에게 즉시 질문하세요. 질문도 검토 프로세스의 일부입니다.
