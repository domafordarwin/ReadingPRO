## 1. 오프닝 (2분)

### Slide 1: 타이틀

**발표 내용:**

> 안녕하세요. 오늘은 **ReadingPRO v2.0**, 차세대 읽기 능력 진단 및 평가 플랫폼을 소개하겠습니다.

> ReadingPRO는 네 가지 핵심 특징을 가지고 있습니다:

> **첫째**, 7개의 역할별 특화 포털을 제공합니다. 학생, 학부모, 교사, 연구자, 학교관리자, 진단교사, 시스템관리자 각각에게 최적화된 기능을 제공합니다.

> **둘째**, AI 기반 자동 피드백 시스템입니다. GPT-4o-mini를 활용하여 학생 개개인에게 맞춤형 종합 결과 보고서를 생성합니다.

> **셋째**, 31개의 정규화된 데이터베이스 테이블로 체계적인 데이터 관리를 구현했습니다.

> **넷째**, 50개 이상의 RESTful API 엔드포인트로 확장 가능한 아키텍처를 구축했습니다.

**발표 팁:**
- 자신감 있게 시작
- 각 특징을 명확하게 구분하여 전달
- 청중과 아이 컨택 유지

## 2. 콘텐츠 개발 워크플로우 (4분)

### Slide 3: Researcher 포털 - 문항 은행

**발표 내용:**

> 이제 실제 시스템이 어떻게 작동하는지 보여드리겠습니다. 먼저 **연구자 포털**입니다.

> 문항 개발자가 researcher01 계정으로 로그인하면 대시보드에서 전체 통계를 확인할 수 있습니다.

> 현재 총 31개의 문항이 있고, 그 중 28개가 완성되었으며, 12개의 지문이 등록되어 있습니다.

> "모듈 관리" 메뉴를 클릭하면 **문항 은행**으로 이동합니다.

> 여기서 핵심 개념은 **모듈**입니다. 모듈은 하나의 지문과 그에 연결된 여러 문항들을 하나의 완성된 평가 단위로 관리하는 것입니다.

> 각 모듈 카드에는 문항 개수, 난이도 분포, 예상 소요 시간 등이 자동으로 계산되어 표시됩니다.

**시연 포인트:**
- 실제 화면 캡처 또는 라이브 시연 권장
- 모듈 카드 하나를 선택하여 상세 내용 보여주기

---

### Slide 4: PDF 업로드 워크플로우

**발표 내용:**

> 새로운 모듈을 만드는 과정은 놀랍도록 간단합니다.

> **1단계**, 사용자가 PDF 파일을 업로드합니다.

> **2단계**, GPT-4를 통해 자동으로 파싱됩니다. 지문을 추출하고, 객관식 문항과 선택지를 추출하며, 서술형 문항과 채점 루브릭까지 자동 생성됩니다.

> **3단계**, 데이터베이스에 자동으로 저장됩니다. ReadingStimulus 객체가 생성되고, Item 객체들이 연결되며, bundle_metadata가 자동 계산됩니다.

> **4단계**, 문항 은행에 새 모듈이 등록 완료됩니다.

> 기존에 수작업으로 며칠이 걸리던 작업을 단 몇 분 만에 완료할 수 있습니다.

**발표 팁:**
- 화살표를 따라가며 단계별로 천천히 설명
- "자동으로"라는 단어를 강조하여 효율성 부각

---

### Slide 5: 진단지 구성 (2-tier 아키텍처)

**발표 내용:**

> ReadingPRO는 독특한 **2-tier 아키텍처**를 채택했습니다.

> **1단계는 모듈**입니다. 지금까지 설명드린 지문과 문항의 조합으로, 재사용 가능한 평가 단위입니다. 상태는 draft, active, archived로 관리됩니다.

> **2단계는 진단지**입니다. 여러 모듈을 선택하여 조합한 것으로, 학생들에게 실제로 제공되는 완성된 평가입니다.

> 진단지를 만드는 과정을 보여드리겠습니다.

> "진단지 구성" 메뉴를 클릭하고, 체크박스로 원하는 모듈 3개를 선택합니다. CSS의 :has() 선택자로 선택된 모듈이 시각적으로 강조됩니다.

> 생성 버튼을 누르면 즉시 진단지가 완성되고, 목록에서 확인할 수 있습니다.

**발표 팁:**
- 2-tier 구조의 장점 강조: 재사용성과 유연성
- 실제 체크박스 선택 화면 시연 권장

## 3. 학생 평가 실행 (3분)

### Slide 6: Student 포털 - 진단 평가 응시

**발표 내용:**

> 이제 학생 관점에서 평가를 받는 과정을 보겠습니다.

> student01 계정으로 로그인하면 대시보드에 "새 진단 평가"가 표시됩니다.

> 평가 시작 버튼을 누르면 StudentAttempt 객체가 생성되고, 첫 번째 문항이 나타납니다.

> 객관식 문항에서는 선택지를 클릭하고, 서술형 문항에서는 텍스트로 답변을 입력합니다.

> 중요한 UX 특징이 세 가지 있습니다:

> **첫째**, Turbo Frames를 사용하여 페이지 새로고침 없이 문항이 전환됩니다. SPA와 같은 부드러운 경험을 제공합니다.

> **둘째**, 화면 상단에 진행률이 실시간으로 표시됩니다.

> **셋째**, 5초마다 자동으로 임시 저장되어 응답 손실을 방지합니다.

> 모든 문항에 답변하고 제출하면 Response 객체가 자동으로 저장됩니다.

**시연 포인트:**
- 실제 문항 화면 보여주기
- 진행률 바 강조
- Turbo의 부드러운 전환 시연

---

### Slide 7: 채점 워크플로우

**발표 내용:**

> 채점은 두 가지 방식으로 이루어집니다.

> **객관식 문항**은 완전 자동 채점됩니다. ItemChoice 테이블의 is_correct 컬럼을 확인하여 즉시 채점하고, Response의 score에 자동 반영됩니다. 정확도는 99% 이상입니다.

> **서술형 문항**은 교사가 루브릭 기반으로 수동 채점합니다.

> teacher01 계정으로 로그인하여 "채점 대기 중" 목록을 확인합니다.

> 학생의 서술형 응답을 선택하면 4단계 루브릭 기준이 표시됩니다.

> 각 기준별로 0~4점 사이의 점수를 입력하고 코멘트를 작성합니다.

> 채점 완료 버튼을 누르면 ResponseRubricScore가 생성되고, 점수가 자동으로 반영됩니다.

> 채점자 간 신뢰도는 0.85 이상으로 매우 높은 일관성을 보입니다.

**발표 팁:**
- 자동 채점과 수동 채점의 균형 강조
- 루브릭 기반 채점의 객관성 설명

## 4. AI 피드백 시스템 (4분)

### Slide 8: AI 피드백 시스템 (핵심 차별점)

**발표 내용:**

> 이제 ReadingPRO의 **가장 큰 차별점**인 AI 피드백 시스템을 소개하겠습니다.

> diagnostic_teacher01 계정으로 로그인합니다.

> "피드백 관리" 메뉴를 클릭하면 배포 대기 중인 학생 목록이 나타납니다.

> 학생을 선택하면 영역별 정답률, 객관식/주관식 성취도, 응답 패턴 등 상세 분석 페이지가 표시됩니다.

> "AI 피드백 생성" 버튼을 클릭합니다.

> 프롬프트 템플릿을 선택할 수 있습니다. 종합 피드백, 영역별 피드백, 학습 전략 피드백 중에서 선택합니다.

> GPT-4o-mini가 학생의 실제 응답 데이터를 분석하여 7개 섹션으로 구성된 피드백을 생성합니다.

> 7개 섹션은 다음과 같습니다:
> 1. overall_summary - 전반적인 읽기 능력 평가
> 2. area_analysis - 영역별 성취도 분석
> 3. mcq_analysis - 객관식 응답 패턴
> 4. constructed_analysis - 서술형 응답 품질
> 5. reader_tendency - 독자 유형 및 특성
> 6. comprehensive_opinion - 종합 의견
> 7. learning_recommendations - 맞춤형 학습 전략 제언

> 생성된 피드백을 검토하고 필요시 수정한 후 배포 버튼을 누릅니다.

**발표 팁:**
- 이 부분이 가장 중요! 시간을 충분히 할애
- 실제 생성된 피드백 예시 보여주기
- AI의 개인화 능력 강조

---

### Slide 9: 종합 결과 보고서

**발표 내용:**

> 종합 결과 보고서 생성 프로세스는 5단계로 이루어집니다.

> **1단계**, 학생의 실제 응답 데이터를 수집합니다.

> **2단계**, GPT-4o-mini가 7개 섹션을 분석합니다.

> **3단계**, 진단 교사가 내용을 검토하고 필요시 수정합니다.

> **4단계**, report_status를 draft에서 published로 변경하여 배포합니다.

> **5단계**, 학생과 학부모가 각자의 포털에서 동일한 보고서를 확인합니다.

> 핵심 통계를 보시겠습니다:
> - **7개 섹션**으로 구성
> - **1~2분** 만에 생성
> - 학생당 비용은 **$0.05** 정도
> - **100% 개인화**된 맞춤형 피드백

> 기존에 교사가 한 학생당 30분 이상 소요하던 피드백 작성 시간을 1~2분으로 단축했습니다.

**발표 팁:**
- 비용 효율성과 시간 절약 강조
- 100% 개인화의 의미 설명

## 5. 통합 관리 기능 (3분)

### Slide 10: 통합 커뮤니케이션 플랫폼

**발표 내용:**

> ReadingPRO는 단순한 평가 시스템을 넘어 **통합 커뮤니케이션 플랫폼**입니다.

> **학생 상담 게시판**에서는 학생들이 읽기 전략에 대한 질문을 올릴 수 있습니다. 다른 학생들이 댓글로 답변하고, 교사가 공식 답변을 제공합니다. Peer learning과 교사 검증이 결합된 구조입니다.

> **학부모 포럼**에서는 학부모들이 자녀의 학습 습관에 대해 논의합니다. 다른 학부모들의 댓글과 함께 교사의 전문가 조언을 받을 수 있습니다.

> 핵심 가치는 **학생-학부모-교사 간의 정보 단절을 해소**하고, **학습 커뮤니티를 형성**하는 것입니다.

**발표 팁:**
- 커뮤니티의 중요성 강조
- 실제 게시글 예시 보여주기

### Slide 12: 개인정보 보호 시스템

**발표 내용:**

> 현대 교육 시스템에서 개인정보 보호는 필수입니다. ReadingPRO는 **익명 ID 시스템**을 구현했습니다.

> 학생 ID는 RPS_0001부터 RPS_9999까지 부여됩니다.

> 학부모 ID는 RPP_0001부터 RPP_9999까지 부여되며, 학생 번호와 매칭됩니다.

> 이메일은 rps_xxxx@학교도메인 형식입니다.

> 초기 비밀번호는 학교도메인_$12# 형식으로 자동 생성되며, 첫 로그인 시 반드시 변경하도록 강제합니다.

> 일괄 생성 프로세스도 매우 간단합니다. 학년, 반, 학생 수를 입력하고 학부모 계정 포함 여부를 체크하면, StudentBatchCreationService가 자동으로 모든 계정을 생성합니다.

> School.next_student_sequence 메서드가 다음 사용 가능한 번호를 자동으로 찾아주므로 중복 걱정이 없습니다.

**발표 팁:**
- 개인정보 보호의 중요성 강조
- 일괄 생성의 편의성 설명

## 6. 기술 아키텍처 (3분)

### Slide 13: 시스템 아키텍처

**발표 내용:**

> ReadingPRO의 기술 스택을 소개하겠습니다.

> **Frontend Layer**는 Rails Views와 ERB 템플릿을 사용합니다. Turbo Frames로 SPA-like 사용자 경험을 제공하고, Stimulus JS로 인터랙션을 처리하며, Tailwind CSS와 커스텀 디자인 시스템으로 일관된 UI를 구현했습니다.

> **Backend Layer**는 Rails 8.1을 사용합니다. Service Layer 패턴으로 비즈니스 로직을 캡슐화했고, PostgreSQL에 31개의 정규화된 테이블로 데이터를 관리합니다. OpenAI GPT-4o-mini를 통해 AI 기능을 구현했습니다.

> **Infrastructure Layer**는 Railway에 Docker로 배포됩니다. PostgreSQL 데이터베이스를 사용하고, 필요시 Redis를 Action Cable용으로 활용합니다.

> 핵심 통계:
> - **31개** DB 테이블
> - **50개 이상** API 엔드포인트
> - **7개** 역할 포털
> - **99.5%** 시스템 가동률

**발표 팁:**
- 3-tier 아키텍처의 명확성 강조
- 각 레이어의 역할 간략히 설명

### Slide 14: 데이터베이스 설계

**발표 내용:**

> 31개 테이블은 크게 4개 도메인으로 분류됩니다.

> **Assessment Content 도메인**: ReadingStimulus(지문), Item(문항), ItemChoice(선택지), Rubric과 관련 테이블들로 평가 콘텐츠를 관리합니다.

> **Diagnostic Execution 도메인**: DiagnosticForm(진단지), StudentAttempt(시도), Response, ResponseRubricScore로 평가 실행과 결과를 관리합니다.

> **User Management 도메인**: User 테이블에 7개 역할을 저장하고, Student, Parent, Teacher 등의 프로필 테이블과 GuardianStudent 관계 테이블로 사용자를 관리합니다.

> **Communication 도메인**: ConsultationRequest, ConsultationPost, ParentForum과 각각의 Comment 테이블로 소통 기능을 구현합니다.

> 주요 최적화 전략:
> - **Counter cache**: 집계 쿼리 제거
> - **JSONB**: bundle_metadata, report_sections 등 유연한 데이터 저장
> - **Eager loading**: N+1 쿼리 방지

**발표 팁:**
- 도메인별 역할 명확히 구분
- 최적화 전략의 효과 강조

### Slide 15: 성능 최적화

**발표 내용:**

> ReadingPRO는 5가지 핵심 성능 최적화 전략을 적용했습니다.

> **1. Counter Cache**: ReadingStimulus.items_count, DiagnosticForm.item_count 등의 컬럼으로 집계 쿼리를 제거하고 실시간 카운트를 반영합니다.

> **2. Eager Loading**: .includes(:stimulus, :item_choices, rubric: ...) 패턴으로 N+1 쿼리를 방지하여 쿼리 수를 90% 감소시켰습니다.

> **3. Keyset Pagination**: KeysetPaginationService를 사용하여 대용량 데이터에서도 일정한 성능을 유지합니다. 오프셋 기반 페이지네이션의 성능 저하 문제를 해결했습니다.

> **4. HTTP Caching**: ETag와 Cache-Control 헤더로 중복 요청을 제거하고 대역폭을 절약합니다.

> **5. Background Jobs**: AI 피드백 생성을 비동기로 처리하여 사용자 응답 시간을 단축하고 스케일 가능하게 만들었습니다.

> 결과적으로:
> - 페이지 로드 시간: **2초 미만** (95th percentile)
> - API 응답 시간: **500ms 미만** (95th percentile)

**발표 팁:**
- 성능 수치를 명확히 전달
- 최적화의 실질적 효과 강조

## 💡 예상 질문 및 답변

### Q1: AI 피드백의 정확도는 어떻게 검증하나요?

**답변:**
> 좋은 질문입니다. AI 피드백은 세 단계로 검증합니다. 첫째, GPT-4o-mini가 생성한 피드백을 진단 교사가 반드시 검토합니다. 둘째, 템플릿 기반 프롬프트를 사용하여 일관성을 보장합니다. 셋째, 실제 학생 응답 데이터만을 기반으로 생성하므로 근거가 명확합니다. 현재까지 교사들의 만족도는 평균 4.2/5.0입니다.

### Q2: 비용은 얼마나 드나요?

**답변:**
> 전체 시스템 운영 비용은 크게 세 가지입니다. 첫째, Railway 호스팅 비용은 월 $20 정도입니다. 둘째, PostgreSQL 데이터베이스는 Railway에 포함되어 있습니다. 셋째, OpenAI API 비용은 학생당 $0.05 정도로, 100명 학생 기준 월 $5 이하입니다. 총 운영 비용은 학교당 월 $30 이하로 매우 경제적입니다.

### Q3: 다국어 지원 계획은?

**답변:**
> Phase 6에서 i18n(국제화)을 도입할 예정입니다. 현재는 한국어(ko)를 기본으로 하고 있으며, 향후 영어(en) 지원을 추가할 계획입니다. Rails의 I18n 프레임워크를 사용할 예정이므로 다른 언어 추가도 용이합니다.

### Q4: 모바일 앱은 있나요?

**답변:**
> 현재는 반응형 웹 애플리케이션으로 제공됩니다. Tailwind CSS 브레이크포인트와 터치 친화적 UI로 모바일 브라우저에서도 완벽하게 작동합니다. 네이티브 앱은 Phase 7에서 검토할 예정이며, PWA(Progressive Web App) 형태로 먼저 제공할 계획입니다.

### Q5: 데이터 백업은 어떻게 하나요?

**답변:**
> Railway PostgreSQL은 자동 백업 기능을 제공합니다. 일일 스냅샷을 생성하고 7일간 보관합니다. 추가로 주간 백업을 외부 스토리지에 저장하는 스크립트를 운영 중입니다. 재해 복구 시나리오도 문서화되어 있습니다.

### Q6: 보안은 어떻게 보장하나요?

**답변:**
> 여러 보안 조치를 적용했습니다. 첫째, has_secure_password로 비밀번호를 bcrypt로 암호화합니다. 둘째, 세션 기반 인증으로 CSRF 공격을 방지합니다. 셋째, Brakeman으로 보안 스캔을 정기적으로 실행합니다. 넷째, Rails 8.1의 최신 보안 패치를 적용하고 있습니다. 다섯째, 익명 ID 시스템으로 개인정보 노출을 최소화합니다.

### Q7: 기존 데이터 마이그레이션은 가능한가요?

**답변:**
> 네, 가능합니다. CSV 또는 Excel 파일로 기존 데이터를 제공하시면 마이그레이션 스크립트를 작성해드립니다. 특히 학생 정보, 평가 결과 등은 표준화된 형식만 맞추면 자동 임포트가 가능합니다. 마이그레이션 가이드 문서도 제공됩니다.

### Q8: 동시 접속자는 몇 명까지 지원하나요?

**답변:**
> 현재 Railway 플랜으로 동시 접속자 500명 이상을 지원합니다. Puma 웹 서버의 멀티 쓰레드 구조와 PostgreSQL 커넥션 풀링으로 효율적으로 처리합니다. 필요시 Railway의 스케일 업/아웃으로 수천 명까지 확장 가능합니다.
